## Context

PD-Spec projects with dense Work layers consume excessive session tokens. Root cause: skills read full data files regardless of how much they actually need. A single `/analyze` run loads ~90,000 tokens of context before doing any real work.

Secondary problem: freemode presentations (HTML monoliths of 250+ KB) require full-file reads for small edits, compounding the token drain.

Both issues are addressed at the engine level, benefiting all current and future projects.

Third finding (session 2026-02-28): The Live Research App's custom Agent Runtime (`agent-runtime.js`) has fundamental limitations that prevent index-based reads and make skills behave differently in CLI vs app. The Claude Agent SDK resolves this — see "Agent Architecture" section below.

### Token consumption breakdown — `/analyze` on pds--timining

| What gets loaded | Size | Tokens (approx) |
|---|---|---|
| CLAUDE.md | 8 KB | ~2,000 |
| analyze/SKILL.md | 36 KB | ~9,000 |
| EXTRACTIONS.md (full read) | 199 KB | **~50,000** |
| INSIGHTS_GRAPH.md (full read) | 68 KB | **~17,000** |
| CONFLICTS.md | 14 KB | ~3,500 |
| MEMORY.md + other Work files | ~30 KB | ~7,500 |
| **Total** | | **~89,000** |

The skill instructions (~10%) are justified — they ensure deterministic execution. The data reads (~90%) are the optimization target.

## BL-101: Work Layer Index System

### Problem

Skills read full accumulation files even when they need a fraction:

| File | Size (pds--timining) | Typical need |
|---|---|---|
| EXTRACTIONS.md | 199 KB (~50K tokens) | 3 of 65 sections (incremental) |
| INSIGHTS_GRAPH.md | 68 KB (~17K tokens) | Scan IDs + titles for dedup |
| _temp/*_normalized.md | 330 KB combined | Reference specific segments |

Synthesis files (STRATEGIC_VISION.md, PROPOSALS.md, RESEARCH_BRIEF.md) are NOT indexable — their value is in the cohesive narrative. They tend to stay under 20 KB and are read in full by design.

### Architecture

**New convention:** `02_Work/_index/` directory for auto-generated indexes.

```
02_Work/
  _index/                              ← auto-generated, safe to delete + regenerate
    EXTRACTIONS_INDEX.md
    INSIGHTS_GRAPH_INDEX.md
    {source}_normalized_INDEX.md
  _temp/                               ← ephemeral session files (unchanged)
    SESSION_CHECKPOINT.md
    {source}_normalized.md
```

**Why `_index/` and not `_temp/`:** The batch processing in `/extract` runs `rm -rf 02_Work/_temp/*` between batches. Indexes need to survive batch boundaries. Separate directory avoids changing existing cleanup logic.

**Threshold:** 20 KB. Files below this are read directly.

**Fundamental rule:** Indexes are an optimization, not a requirement. If the index exists and is fresh → skill reads index. If missing or stale → skill falls back to reading the full file (current behavior). No skill should DEPEND on the index.

### Index Formats

Each file type needs a different structure because the underlying data is different.

#### EXTRACTIONS_INDEX.md

Purpose: Let skills identify which sections exist, when they were extracted, and jump to specific claims by line number.

```markdown
# Extractions Index
> Auto-generated by generate-index.sh. Do not edit manually.
> Source: 02_Work/EXTRACTIONS.md
> Generated: 2026-02-27T18:00
> Source lines: 2114 | Source hash: a1b2c3 (first 8 chars of md5)
> Sections: 65 | Total claims: 1516

| Section | Type | Extracted | Claims | L-start | L-end |
|---|---|---|---|---|---|
| entrevistas operaciones/analisis reu con operaciones.md | meeting_analysis | 2026-02-16T15:00 | 40 | 8 | 55 |
| entrevistas operaciones/reu_coo_roberto-catalan.md | coo_meeting | 2026-02-16T15:00 | 25 | 56 | 87 |
| ... | ... | ... | ... | ... | ... |
```

**Key design decisions:**
- Header includes source file hash for staleness detection
- No per-claim summaries — section-level granularity (65 rows) keeps the index under 5 KB while enabling targeted reads via `L-start`/`L-end`
- Per-claim index would be ~30 KB (1,500 rows) — defeats the purpose

#### INSIGHTS_GRAPH_INDEX.md

Purpose: Let skills scan all insights by ID/title/status/convergence without loading full narratives and evidence trails.

```markdown
# Insights Graph Index
> Auto-generated by generate-index.sh. Do not edit manually.
> Source: 02_Work/INSIGHTS_GRAPH.md
> Generated: 2026-02-27T18:00
> Source lines: 1200 | Source hash: d4e5f6
> Total: 47 | VERIFIED: 39 | PENDING: 0 | MERGED: 1 | INVALIDATED: 7

| ID | Title | Status | Category | Convergence | Voice | L-start |
|---|---|---|---|---|---|---|
| IG-01 | 6 productos en 2 dominios: Ops y Geo | VERIFIED | product-landscape (current) | 4/21 | stakeholder | 15 |
| IG-02 | Aware+Orchestra solución integrada | VERIFIED | product-landscape (current) | 3/21 | stakeholder | 32 |
| IG-SYNTH-01 | Geometry Gap temporal | VERIFIED | user-need (current) | 6/21 | user, stakeholder | 245 |
| ... | ... | ... | ... | ... | ... | ... |
```

**Key design decisions:**
- One row per insight — at 47 insights this is ~5 KB, at 200 insights ~20 KB
- Title is the full insight title (sufficient for semantic dedup by LLM)
- `L-start` enables targeted read of full entry (~15-20 lines each)
- Header stats match `count-statuses.sh` output — cross-validation built in

#### _normalized_INDEX.md

Purpose: Let skills reference specific parts of normalized transcripts without loading 60-130 KB files.

```markdown
# Normalized File Index
> Auto-generated by generate-index.sh. Do not edit manually.
> Source: 02_Work/_temp/reunion_camila_normalized.md
> Generated: 2026-02-27T18:00
> Source lines: 1400 | Source hash: g7h8i9
> Speakers: Camila, Nicolás | Duration: ~45 min (estimated)

| Section/Topic | Speaker(s) | L-start | L-end |
|---|---|---|---|
| Contexto proyecto TIMining | Camila | 1 | 85 |
| Productos actuales y dolor | Camila, Nicolás | 86 | 210 |
| Feedback touchpoint 1 | Camila | 211 | 340 |
| ... | ... | ... | ... |
```

**Indexing strategy:** Index by markdown headers (if normalization added them). Fallback to fixed-size chunks (100 lines) if no headers. Semantic topic segmentation deferred — revisit if header/chunk indexing proves insufficient.

### Script: generate-index.sh

New utility script. Follows the 90/10 rule — 100% mechanical (parsing headers, counting lines, computing hashes).

```
Usage: ./scripts/generate-index.sh <type> <source-file> [output-file]

Types:
  extractions    Parse ## [section] headers, metadata, claim counts
  insights       Parse ### [IG-XX] entries, status, category, convergence
  normalized     Parse ## headers or chunk by 100-line blocks

If output-file is omitted, writes to 02_Work/_index/{basename}_INDEX.md

Exit codes:
  0  Index generated successfully
  1  Source file not found
  2  Source file below threshold (< 20KB) — no index needed
```

**Implementation:** bash + awk/grep. No LLM, no Node.js.

### Staleness Detection

**Mechanism:** MD5 hash (first 8 chars) stored in index header.

Before using an index, the skill runs:
```bash
current_hash=$(md5 -q 02_Work/EXTRACTIONS.md | cut -c1-8)
```
If `current_hash != index_hash` → stale → regenerate or fallback.

**Cost:** One shell command (~50ms). Zero tokens.

**Mid-execution staleness:** If a skill reads the index at start, then writes to the source file, the index becomes stale. This is expected — the skill already has the data in context. It regenerates the index at the end.

### Skill Modifications

**⚠️ Runtime-agnostic constraint:** After BL-80 Phase 1, skills run identically in Claude Code CLI and the Agent SDK in the app. SKILL.md instructions must describe *what to do*, not *which tool to use*:

| ✗ Don't write | ✓ Write instead |
|---|---|
| "Use the Bash tool to run `md5 -q`" | "Run `./scripts/generate-index.sh`" |
| "Use Read with offset=X, limit=Y" | "Read lines X–Y of the file" |
| "Call the Grep tool to search" | "Search the file for pattern X" |

The agent resolves these to the correct tool in each runtime.

#### /extract → generates EXTRACTIONS index + normalized indexes
After Phase 2 completion: run `generate-index.sh extractions` and `generate-index.sh normalized` for each file > 20 KB.

#### /analyze → consumes EXTRACTIONS index + INSIGHTS_GRAPH index, generates INSIGHTS_GRAPH index
- Phase 0: Read indexes instead of full files (with hash validation + fallback)
- Phase 1 (incremental): Filter sections via index table (timestamps already there)
- Phase 2 (dedup): Compare claims against index titles. Only read lines L-start through L-start+20 of INSIGHTS_GRAPH.md for potential matches.
- After Phase 4: Regenerate insights index.

#### /spec → consumes INSIGHTS_GRAPH index, regenerates after modifications
#### /audit, /ship, /visualize → consume indexes (read-only, no regeneration needed)

### Estimated Token Savings

| Scenario | Current | With indexes | Savings |
|---|---|---|---|
| `/analyze` incremental (3/65 sections) | ~89K tokens | ~15K tokens | **~83%** |
| `/analyze --full` | ~89K tokens | ~89K tokens | 0% (reads everything) |
| `/audit` (read-only scan) | ~75K tokens | ~12K tokens | **~84%** |
| `/ship prd` | ~75K tokens | ~15K tokens | **~80%** |
| Freemode referencing EXTRACTIONS | ~50K tokens | ~5K tokens | **~90%** |

### Gaps & Decisions for Implementation

1. **CONFLICTS.md** (14 KB, below threshold) — apply same pattern if/when it crosses 20 KB. No preemptive work.
2. **SOURCE_MAP.md** (8.4 KB) — unlikely to cross threshold. No action.
3. **Live Research App (`/view`)** — has its own parsers, doesn't use Claude Code Read tool. No changes needed.
4. **Files oscillating around threshold** — `generate-index.sh` returns exit code 2 if < 20 KB. Skill handles gracefully.

## BL-102: Showcase — Presentation System

> **Scope updated 2026-02-28:** Full scope (XL) reduced to v1 (L) focused on index.html migration. v2/v3 deferred.

### Problem

Current freemode presentations are HTML monoliths (index.html = 253 KB, styles.css = 68 KB). Content, layout, and styles are tangled. Editing one slide requires reading 250+ KB (~63,000 tokens).

### Architecture

**Separation of concerns:**

```
Content (MDX slides)     →  what to say       (~2 KB per slide)
Engine  (Astro app)      →  how to present it  (components, layouts)
Theme   (CSS tokens)     →  how it looks       (design tokens)
```

**Repository structure:**

```
pd-showcase/                        ← SEPARATE REPO (git submodule of pd-spec)
  src/
    components/
      primitives/                   ← v1: Grid, Card, Image, Quote, Stat, Tag, Icon
      patterns/                     ← v1: BenchCard, CaseStudy, DarFlow, RecBox, LayerDiagram
    layouts/
      SlideLayout.astro             ← single slide wrapper
      PresentationLayout.astro      ← deck wrapper (navigation, counter, transitions)
    themes/
      theme_spec.css                ← default theme (extracted from existing styles.css)
    pages/
      [deck]/
        index.astro
        [...slide].astro            ← dynamic slide routing
    content/
      config.ts                     ← Zod schemas
  astro.config.mjs                  ← reads content from external dir
  package.json
```

**In each pd-spec project:**

```
showcase/                           ← git submodule → pd-showcase
03_Outputs/_showcase/               ← project-specific content
  decks/
    main/                           ← v1: single deck only
      cover.mdx
      contexto.mdx
      pilares.mdx
      ...
  theme/
    overrides.css                   ← project theme overrides (v1: optional)
  public/
    img/                            ← images and assets
  deck.config.ts                    ← deck metadata (brand, language)
```

**Content outside submodule:** Astro 5 content layer with `glob()` loader reads from `../../03_Outputs/_showcase/decks`. Submodule = pure engine. Content = project-specific.

### Phasing

#### v1 — Single Deck Migration (Effort: L) ← THIS PLAN

Scope: Migrate pds--timining's `index.html` (35 slides) to MDX.

**7 components** (derived from slide-by-slide analysis of index.html):

### Content Schema

#### Slide Frontmatter (Zod)

```typescript
const slideSchema = z.object({
  order: z.number(),                          // position within deck
  title: z.string(),
  subtitle: z.string().optional(),
  layout: z.enum([
    'cover', 'section', 'content', 'grid',
    'compare', 'bench', 'image-full', 'quote', 'custom'
  ]).default('content'),
  notes: z.string().optional(),               // speaker notes (presenter view)
  tags: z.array(z.string()).optional(),        // ["new", "updated"]
  refs: z.array(z.string()).optional(),        // [IG-XX] traceability
  hidden: z.boolean().default(false),          // skip in presentation, keep in repo
});
```

**Design decisions:**
- `order` in frontmatter, not filename — reordering = change a number
- `layout` is a hint, not a constraint — MDX body always wins
- `refs` enables slide-level traceability to insights

#### Example Slide

```mdx
---
order: 2
title: "Lo Que Escuchamos"
layout: "grid"
tags: ["new"]
refs: ["IG-26", "IG-27", "IG-28", "IG-29"]

import { Grid, Card, Icon } from '@showcase/primitives'

<Grid cols={4}>
  <Card>
    <Icon name="clock" />
    ### Tiempo, no paz mental
    Carlos: "Es un mundo sin tiempo para avanzar."
    <InsightRef id="IG-26" />
  </Card>
  <!-- ... more cards -->
</Grid>
```

**Token cost:** ~700 tokens (read + edit) vs ~63,000 tokens for equivalent HTML edit.

## BL-80: Agent Architecture — SDK Migration

> **Added 2026-02-28.** This section was not part of the original plan. It emerged from analyzing the App Runtime Compatibility Gap in BL-101 and realizing the custom Agent Runtime is the wrong foundation for scaling.

### Problem

The Live Research App's custom Agent Runtime (`agent-runtime.js`) reimplements a fraction of Claude Code's capabilities:

| Capability | Custom Runtime | Claude Agent SDK |
|---|---|---|
| File reads | `read_file` — full file only | `Read` — with offset + limit |
| Shell | `run_script` — 6 whitelisted scripts | `Bash` — full shell (sandboxed) |
| File search | `search_files` — 5-match limit | `Grep` + `Glob` — full capabilities |
| Context mgmt | Manual trimming to 2000 chars | Automatic compaction |
| Model | Hardcoded Sonnet | Configurable per-query |
| Subagents | Not supported | `Task` tool native |
| Sessions | In-memory, 8h TTL | Persistent with resume/fork |

Skills cannot run identically in CLI and app. Every engine improvement (indexes, new skills) requires parallel compatibility work.

### Solution: Claude Agent SDK

Replace the custom runtime with the Claude Agent SDK (`@anthropic-ai/claude-agent-sdk`). The SDK provides `query()` — an async generator that streams the full agent loop with all built-in tools.

Key integration point: `canUseTool` callback. When the agent needs user interaction, the callback sends the request to the frontend via WebSocket, pauses execution, and waits for the user's response. This replaces the custom `ask_confirmation`/`ask_selection`/`ask_text` tools.

### Service Abstractions (SaaS-ready, localhost implementations)

Phase 1 defines interfaces designed for multi-tenant SaaS, with simple implementations for localhost:

| Abstraction | Phase 1 (localhost) | Phase 3 (SaaS) |
|---|---|---|
| `WorkspaceService` | Local filesystem path | Isolated container per user/project |
| `SessionStore` | In-memory Map | Database (Postgres/Redis) with TTL |
| `ExecutionQueue` | Direct `query()` call | Job queue with concurrency limits |
| `KeyVault` | Memory (current BYOK) | Encrypted store, BYOK + platform keys |

The SDK is called through these abstractions — never directly from route handlers.

### Strategic Fork: SaaS vs MCP App

Two distribution models (not mutually exclusive):

| | Route A: SaaS standalone | Route B: MCP App |
|---|---|---|
| User goes to | pd-spec.app (your domain) | Claude, ChatGPT, VS Code, any MCP host |
| UI ownership | Full control (React frontend) | MCP Apps protocol (iframed components) |
| Infrastructure | You own servers, auth, billing | Host provides infra |
| Multi-host | Only your app | Any MCP-compatible client |

**Risk for Route B:** MCP Apps is new (Jan 2026). May not gain host adoption. Complex multi-step UX may not fit in chat-embedded iframes. Counter-argument: the target is accessibility, not power users.

**Decision point:** Evaluate before Phase 3. Phases 1-2 are route-agnostic. By Phase 3, MCP Apps maturity and host adoption will be clearer.

### Impact on BL-101

The index system's App Runtime Compatibility Gap **disappears** after BL-80 Phase 1. The SDK has `Read` with offset/limit and `Bash` natively. `generate-index.sh` executes without whitelisting. Skills work identically in both environments.

BL-101 can be implemented immediately for CLI. Full app compatibility arrives with BL-80 Phase 1.
