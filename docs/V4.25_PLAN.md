## Decision 1: BL-92 — Execution Model

### Hybrid: scripts for mechanical, agent for semantic

```
                    ┌─────────┐
                    │ Scripts  │  ← shared execution layer
                    └────┬────┘
                         │
              ┌──────────┼──────────┐
              │                     │
         ┌────┴────┐          ┌────┴────┐
         │   App   │          │  Skills │
         │ (UI)    │          │ (agent) │
         └─────────┘          └─────────┘
          primary              fallback / power user
```

- **Scripts** = single source of truth for mechanical operations. Called by BOTH app and skills.
- **App** = primary UI. Phase 1 read-only. Phase 2 calls scripts for writes. Phase 3 calls Claude API for semantic.
- **Skills** = NOT deprecated. Slimmed down (call scripts for mechanical steps). Fallback when app fails.

### Skill Classification

| Skill | Decision | Mechanical → script | Semantic → LLM stays |
|---|---|---|---|
| `/extract` | **Slim down** | Discovery, hash, delta, Pass A+B | Pass C sentence repair, claim extraction |
| `/analyze` | **Slim down** | Phase 1 (load, filter, count) | Phase 2 (dedup, categorize), Phase 3 (synthesis) |
| `/spec` | **Slim down** (was /resolve) | Status changes via scripts | Strategic Vision narrative, Design Proposals |
| `/ship` | **Keep as-is** | (minimal) | All 10 output types (90% semantic) |
| `/audit` | **Slim down** | 7+ scoring checks (grep-based) | Score interpretation, remediation narrative |
| `/visualize` | **Migrate to app** | — | App renders Mermaid directly |
| `/kickoff` | **Expand (future)** | — | Future: Source Setup Wizard |
| `/seed` | **Keep as-is** | — | Content generation (85% semantic) |
| `/reset` | **Script only** | 100% → `./scripts/reset.sh` | — |
| `/view` | **Script only** | 100% → `./scripts/view.sh` | — |

## Decision 2: BL-73 + BL-95 — Strategic Vision + Design Proposals

### Two files replace SYSTEM_MAP.md

**Conceptual tree** (documented in FRAMEWORK.md):

| Layer | Purpose | Entities | PD-Spec file |
|---|---|---|---|
| **Strategic** (why) | Vision, strategy, principles | Vision, Strategy, Design Principles, Domains, Value Props | `STRATEGIC_VISION.md` (Work) |
| **Structural** (what) | What to build | Domain > Module > Feature | `PROPOSALS.md` (Work) |
| **Behavioral** (how user interacts) | Use cases, stories | Use Case, Scenario, User Story, AC | `03_Outputs/` (future /ship types) |
| **Materialization** (how it looks) | Flows, patterns, screens | User Flow, UI Pattern, Screen, Component | Out of scope (PD-Build) |

**Design Principles are transversal** — not a tree node but a filter that crosses all layers. Each principle declares which layers it operates at.

### `STRATEGIC_VISION.md` structure

```markdown
# Strategic Vision

## Product Vision
> Aspirational north. One sentence.

## Product Strategy
> How you get there: market, differentiation, model.

## Design Principles
> Transversal decision criteria. Each declares where it operates.
### [principle name]
- **Operates at:** Strategic / Structural / Behavioral / Materialization
- **Description:** ...
- **Refs:** [IG-XX]

## Operational Domains
> High-level functional territory groupings.
### [domain name]
- **Lead pillar:** [principle name]
- **Refs:** [IG-XX]

## Value Propositions
> User outcomes.

## Open Questions
```

### `PROPOSALS.md` structure

```markdown
# Design Proposals

## Domain: [domain name]

### [DP-XX] [Name]
**Type:** module | feature
**Domain:** [domain name]
**Status:** PROPOSED | VALIDATED | BUILDING | SHIPPED
**Grounded in:** [IG-XX], [IG-XX]
**Parent:** [DP-XX] (features point to their module)
**Design implications:**
- ...
**Acceptance criteria:**
- ...
**Open questions:**
- ...
```

### `/spec` skill changes (was /resolve)

| Before | After |
|---|---|
| Writes SYSTEM_MAP.md | Writes STRATEGIC_VISION.md + PROPOSALS.md |
| Resolves conflicts only | Pre-work: resolves conflicts if pending, then writes spec |
| One parser | Two parsers (strategic-vision.js + proposals.js) |

### Conflict resolution model

- `/spec` resolves pending conflicts as pre-work (semantic judgment by LLM)
- Scripts handle mechanical status changes (`resolve-conflict.sh`)
- App buttons (Phase 2) call scripts for one-off resolutions
- User can resolve individual conflicts without running full `/spec`

## Decision 3: BL-93 + BL-87 — Insight Lifecycle

### 6 statuses + 2 visual flags + 4 implementation layers

### State Machine

```
                        /analyze creates
                              │
                              ▼
                        ┌──────────┐
                        │ PENDING  │
                        └────┬─────┘
                             │
              ┌──────────────┼──────────────┐
              │ verify       │ invalidate   │ merge
              ▼              ▼              ▼
        ┌──────────┐  ┌────────────┐  ┌──────────┐
        │ VERIFIED │  │INVALIDATED │  │  MERGED  │
        └────┬─────┘  │ + reason   │  │→ [IG-XX] │
             │        └────────────┘  └──────────┘
        ┌────┼────────────┐
     freeze  │         supersede
        ▼    │             ▼
   ┌────────┐│       ┌────────────┐
   │ FROZEN ││       │SUPERSEDED  │
   └───┬────┘│       │→ [IG-XX]   │
    unfreeze │       └────────────┘
        ▼    │
   (VERIFIED)│
```

### 6 Statuses

| Status | Meaning | Convergence? | Reversible? |
|---|---|---|---|
| `PENDING` | New, unverified | Yes | — |
| `VERIFIED` | Confirmed by evidence | Yes | — |
| `FROZEN` | Valid but deprioritized ("no mueve la aguja") | **No** | Yes → VERIFIED |
| `INVALIDATED` | Contradicted + reason stored | No | Rare |
| `MERGED` | Absorbed into [IG-XX] | No | No |
| `SUPERSEDED` | Replaced by newer [IG-XX] | No | No |

### 2 Visual Flags (not statuses)

| Flag | Mechanism | What it shows |
|---|---|---|
| Freshness (green/yellow/red) | Parser: `age_in_sessions` since last evidence | How old |
| Challenged (warning) | Grep: open conflict referencing this insight | Has dispute |

### Transitions: who judges, who executes

| Transition | Judgment | Execution |
|---|---|---|
| PENDING → VERIFIED | LLM or user | `verify-insight.sh` |
| PENDING → INVALIDATED | LLM or user + reason | `verify-insight.sh --invalidate --reason "..."` |
| PENDING → MERGED | LLM (dedup in /analyze) | `verify-insight.sh --merge --target IG-XX` |
| VERIFIED → FROZEN | **User only** (no LLM) | `verify-insight.sh --freeze` |
| VERIFIED → SUPERSEDED | LLM (subsumption in /analyze) | `verify-insight.sh --supersede --target IG-XX` |
| VERIFIED → INVALIDATED | LLM or user + new evidence | `verify-insight.sh --invalidate` |
| FROZEN → VERIFIED | **User only** (unfreeze) | `verify-insight.sh --unfreeze` |

### Cascade Protection

Before FREEZE or INVALIDATE, `verify-insight.sh` calculates impact:
- Greps `[IG-XX]` across STRATEGIC_VISION + PROPOSALS + Outputs
- LOW (<=2 refs): proceed silently
- MEDIUM (3-5 refs): show orphaned refs, ask confirm
- HIGH (5+ refs or design principle): show full tree, require explicit confirm

### 4 Implementation Layers

| Layer | What | v4.25? | Mechanism |
|---|---|---|---|
| L1: Freshness | Visual age indicator per insight. New `Last-updated: YYYY-MM-DD` field in INSIGHTS_GRAPH format. | **Yes** | Parser + app component |
| L2: Supersession | /analyze detects newer insight replacing older | No | Extends /analyze Phase 2 |
| L3: Relevance scoring | Composite: freshness x convergence x references | No | Needs L1+L2 |
| L4: Human curation | Freeze/unfreeze with cascade protection | **Partial** — script yes, app button no | Script in v4.25, UI in v4.26 |

## Additional Decisions

### `/resolve` renamed to `/spec`

Pipeline: `/extract` → `/analyze` → `/spec` → `/ship`

Rationale: "resolve" implied the skill only resolves conflicts. The new scope (Strategic Vision + Design Proposals + conflict resolution) is specification work — hence `/spec`.

### Conceptual Tree

4-layer product specification model documented in FRAMEWORK.md. PD-Spec operates at layers 1-2 (Strategic + Structural). Layer 3 (Behavioral) is partially covered by `/ship` output types. Layer 4 (Materialization) is PD-Build territory.
